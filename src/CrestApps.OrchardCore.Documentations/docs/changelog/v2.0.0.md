# Version 2.0.0 Release Notes

**Package version**: `2.0.0-preview-0003`

Version `2.0.0` is not just another incremental update. It is the moment CrestApps.OrchardCore grows into a much larger platform for Orchard Core users who want modern AI, agentic automation, external integrations, document intelligence, voice experiences, omnichannel communication, and schema-driven configuration all in one ecosystem.

This release brings a huge amount of power to Orchard Core. It turns the platform into a place where you can build simple AI chat experiences, rich Copilot-style assistants, tool-using agents, document-aware solutions, MCP-enabled integrations, A2A-connected multi-agent systems, and live-agent handoff flows without having to assemble everything from scratch yourself.

If you are migrating from `v1.x`, expect breaking changes across architecture, configuration, deployment modeling, orchestration, and module behavior. This page intentionally does **not** try to list every single breaking change from the `v1.x` line because the scope of change is simply too large. Instead, it highlights the most important platform capabilities introduced in this branch and points you to the right documentation for deeper details. For deployment migration details, start with the [typed deployments migration guide](../ai/migration-typed-deployments.md).

---

## What changed in 2.0.0

At a high level, `2.0.0` introduces a new generation of CrestApps.OrchardCore built around:

- orchestrator-driven execution instead of simple pass-through prompt handling
- first-class AI deployments and provider connections
- reusable AI templates and agent definitions
- ad-hoc chat interactions and Copilot-style chat experiences
- bring-your-own-data and document-based retrieval across multiple providers
- memory, MCP, and A2A integrations
- extensible response handling for external chat systems and live-agent transfer
- omnichannel communications
- expanded recipe support and schema-driven recipe authoring

For a broad product tour, see the [Artificial Intelligence Suite overview](../ai/index.md) and [Omnichannel Communications](../omnichannel/index.md).

## Major platform shifts

### Orchestrator-based AI execution

The biggest architectural change in `2.0.0` is the move away from a simpler "send prompt to provider, receive message back" model and toward a much more capable orchestrator-based execution model.

The new orchestrator pipeline can plan work, scope tools, resolve agents, inject context, coordinate retrieval, and iterate across multiple steps when a task requires more than one pass. This opens the door to richer agentic behavior, multi-step workflows, smarter prompt handling, preemptive retrieval, and more advanced real-world task execution patterns that simply were not practical in the earlier direct-provider approach.

This is one of the core reasons `2.0.0` feels so much more powerful: the system is no longer just answering prompts, it can actively coordinate how a request should be handled.

Learn more in [Artificial Intelligence Suite](../ai/index.md), [AI Tool Management](../ai/tools.md), and [AI Agent](../ai/agent.md).

### AI services as a full platform

`CrestApps.OrchardCore.AI` is now a much broader foundation instead of a thin AI chat wrapper. It provides a real platform for building AI-powered Orchard Core solutions with deployment management, multiple profile types, template support, workflow integration, voice modes, analytics, response handlers, post-session processing, and configuration-driven defaults across the admin UI.

This release is exciting because it gives Orchard Core users a serious toolbox. You are no longer limited to one narrow AI scenario. You can mix chat, agents, tools, workflows, documents, data sources, integrations, and external protocols into one cohesive solution.

Deployments are also now part of the base `CrestApps.OrchardCore.AI` feature instead of a separate `CrestApps.OrchardCore.AI.Deployments` feature. That means deployment management, deployment endpoints, default deployment settings, and deployment-aware profile editors come online with the main AI feature, which simplifies setup and avoids feature-ordering issues for modules that depend on deployments.

Deployments can now also advertise more than one capability type at the same time. A single deployment such as `gpt-4.1-mini` can be marked for both Chat and Utility, the deployment editor now uses per-type checkboxes, and resolver/default-selection logic treats one deployment as eligible for every selected capability.

The AI Profile and AI Chat Interaction admin experiences are also now source-agnostic. Creating a profile or starting a new chat interaction no longer requires picking a source first; the editor opens directly and the selected deployments determine the active client/model behavior.

Explore [Artificial Intelligence Suite](../ai/index.md), [AI Chat](../ai/chat.md), and [Consuming AI Services via Code](../ai/consuming-ai-services.md).

### Response handlers and third-party chat integration

Chat handling is now extensible through pluggable response handlers, making it possible to route a conversation to AI, a deferred external system, or a live-agent workflow. This enables integrations such as external chat platforms, agent handoff, webhook-driven deferred responses, and persistent relay-based communication.

This is one of the most important practical advancements in the release because it means Orchard Core chat is no longer locked into a single response path. You can build customer support flows, escalation paths, hybrid AI-and-human experiences, and deeply integrated external communication scenarios.

Response-handler-driven assistant messages can now also override the default bot presentation by supplying a custom icon, a CSS color class such as `text-secondary`, and an option to disable the streaming spinner animation. This makes live-agent and transferred conversations look like real people in the transcript instead of always appearing as the green AI bot.

The response-handler API now exposes this customization through the typed `ChatResponseHandlerContext.AssistantAppearance` property, and the hubs persist or return `AssistantMessageAppearance` directly through `Put(...)` and `As<AssistantMessageAppearance>()` instead of a separate helper. The response-handler documentation also now makes it explicit that deferred webhook implementations must persist `AssistantMessageAppearance` on the saved assistant prompt and include it in `ReceiveConversationAssistantToken(...)` if they want the appearance to survive reloads and be broadcast correctly to connected clients. `AssistantMessageAppearance` can also override the visible assistant label, so live-agent and external-handler replies can display names such as `Mike` or `Agent` instead of always showing `Assistant`.

See [Response Handlers](../ai/response-handlers.md) and [Chat UI Notifications](../ai/chat-notifications.md).

## New modules and features

### AI Templates

AI templates are now a first-class feature. The platform supports reusable prompt templates and reusable profile templates so teams can standardize system prompts, instructions, deployments, capabilities, and profile configuration across multiple chat experiences.

This is a major quality-of-life improvement for teams because it becomes much easier to scale repeatable AI experiences across tenants, environments, teams, and use cases. Instead of copying prompt logic everywhere, you can define reusable building blocks and keep your AI behavior more consistent and easier to maintain.

See [AI Prompt Templates](../ai/prompt-templates.md) and [AI Profile Templates](../ai/profile-templates.md).

### AI Chat Interactions

AI Chat Interactions provide ad-hoc, highly configurable chat sessions without requiring a predefined AI profile. Users can adjust behavior at runtime, attach documents, upload images, generate charts, and work with more interactive, session-specific experiences.

This makes the platform feel dramatically more dynamic. It opens the door to guided assistants, temporary session-based workflows, richer front-end AI experiences, and interactions where the user can bring their own context into the conversation instead of relying only on static configuration.

See [AI Chat Interactions](../ai/chat-interactions.md).

### AI Chat Copilot Integration

The new Copilot integration adds a GitHub Copilot-style chat experience and a Copilot-oriented orchestration layer for AI completions.

For users who want a more modern assistant experience directly inside Orchard Core, this is a very exciting addition. It helps bring the feel of contemporary AI tooling into the CMS while still taking advantage of Orchard Core content, services, tools, and integrations.

See [AI Chat Copilot Integration](../ai/copilot.md).

### AI Data Sources

AI Data Sources are now a first-class platform capability. In earlier versions, bring-your-own-data scenarios were effectively tied to Azure OpenAI. In `2.0.0`, external data indexing and retrieval are modeled independently so you can bring your own data while using different AI providers, not just Azure OpenAI.

This is a major unlock for Orchard Core users because your data strategy is no longer boxed into a single provider story. You can design AI experiences around your own data and your preferred provider model, which gives far more freedom for enterprise architecture, cost management, and long-term flexibility.

See [AI Data Sources](../ai/data-sources/index.md), [Azure AI data sources](../ai/data-sources/azure-ai.md), and [Elasticsearch data sources](../ai/data-sources/elasticsearch.md).

### AI Documents

AI Documents introduces a dedicated document pipeline for upload, parsing, chunking, indexing, and retrieval. This includes support for PDF and OpenXML formats and provider-specific indexing backends.

This is where the platform becomes especially compelling for knowledge-heavy solutions. You can turn documents into usable AI context, power document-aware assistants, and build systems that answer questions grounded in real files rather than generic model knowledge alone.

See [AI Documents](../ai/documents/index.md), [PDF support](../ai/documents/pdf.md), and [OpenXML support](../ai/documents/openxml.md).

### AI Memory

AI Memory adds persistent, user-scoped memory so the system can retain durable context across sessions. It integrates with orchestration, retrieval, and provider-backed indexing to support longer-lived, personalized AI experiences.

This is one of the features that moves the platform from isolated chat responses to a more intelligent, evolving assistant model. It helps create experiences that feel more aware, more personalized, and more useful over time.

See [AI Memory](../ai/memory.md).

### MCP Server

The platform now supports exposing Orchard Core as an MCP server so external AI clients can discover and use Orchard-hosted tools, prompts, and resources.

This is exciting because it allows Orchard Core to become part of the wider AI tooling ecosystem, not just a closed application. Orchard can now expose valuable capabilities outward and participate in modern model-context workflows.

See [MCP Server](../ai/mcp/server.md) and the broader [Model Context Protocol overview](../ai/mcp/index.md).

### MCP Clients

`2.0.0` also introduces MCP client capabilities so Orchard Core AI experiences can connect to external MCP servers and consume their tools and resources.

This gives Orchard Core users access to a growing ecosystem of external tools and context providers. It becomes possible to extend Orchard AI experiences far beyond what is built locally by plugging into remote MCP services as part of the orchestration flow.

See [MCP Client Integration](../ai/mcp/client.md) and [Model Context Protocol overview](../ai/mcp/index.md).

### Agent-to-Agent (A2A)

Support for the A2A protocol allows Orchard Core to both expose local agents and connect to external A2A hosts. This enables multi-agent interoperability and distributed agent architectures.

This is one of the most forward-looking features in the release. It lets Orchard Core participate in distributed agent ecosystems where local and remote agents can cooperate, specialize, and expand what your solution can do.

See [Agent-to-Agent Protocol (A2A)](../ai/a2a/index.md), [A2A Host](../ai/a2a/host.md), and [A2A Client Integration](../ai/a2a/client.md).

### AI Agent

AI Agent support adds reusable agent-style profiles and agent-oriented execution patterns. Agents can be exposed locally, used inside orchestration, and made available through A2A and other integrations.

This gives builders a much more expressive way to define specialized AI behavior. Instead of a single general-purpose assistant, you can compose solutions out of focused agents that each have their own role, tools, and purpose.

See [AI Agent](../ai/agent.md) and [AI Tool Management](../ai/tools.md).

### Recipes Module

The Recipes module now plays an important role in the new platform by providing richer AI-related recipe support and schema-driven recipe authoring. The branch also adds authoritative recipe schema generation to improve automation and reduce invalid AI-generated recipe JSON.

This is especially important for users who want reliable automation. Better schemas mean better guidance for AI-generated recipes, fewer invalid configurations, and a smoother path for setup, deployment, and repeatable environment provisioning.

See [Recipes Module](../modules/recipes.md).

### Omnichannel Communications

Omnichannel Communications is now a full feature area rather than a partial experiment. It provides the foundation for multi-channel communication workflows, including management experiences and channel-specific integrations such as SMS and Event Grid.

This is an exciting direction for Orchard Core users who want to go beyond traditional CMS boundaries. It helps Orchard participate in real communication flows across different channels and makes the platform more useful for customer engagement, notifications, and external event-driven solutions.

See [Omnichannel Communications](../omnichannel/index.md), [SMS](../omnichannel/sms.md), [Event Grid](../omnichannel/event-grid.md), and [Management](../omnichannel/management.md).

## Key improvements across the platform

### AI profiles and profile capabilities

AI Profiles are significantly more capable in `2.0.0`, and they are now one of the best examples of how much the platform has matured.

- Profiles now support richer profile types, including agent-oriented scenarios.
- Profiles can work with MCP connections and A2A agents.
- Profiles support initial prompt behavior so the AI can proactively start the conversation instead of waiting for the user to send the first message.
- Chat experiences can generate charts and images when the necessary capabilities and deployments are configured.

This makes profiles far more expressive and much closer to real product design tools. You can shape how an assistant behaves, what tools it can access, how it should start, and what kinds of experiences it can deliver.

See [AI Chat](../ai/chat.md), [AI Agent](../ai/agent.md), [AI Tool Management](../ai/tools.md), and [Profiles in code](../ai/profiles-code.md).

### Typed AI deployments and connection model

AI deployments are now first-class, typed entities instead of being loosely embedded into provider connection settings. This is one of the most important changes in the release and drives the new configuration model across profiles, templates, voice features, and orchestration. The typed deployment migration also backfills the site-level `DefaultAIDeploymentSettings` defaults from the best available deployments when those settings are still unset, which helps upgraded tenants keep working global fallbacks without overwriting administrator choices.

The result is a cleaner, more understandable platform model that better reflects how modern AI systems are actually built. It is easier to reason about which deployment powers which capability and easier to manage defaults across the solution.

See [Migrating to Typed AI Deployments](../ai/migration-typed-deployments.md) and [Artificial Intelligence Suite](../ai/index.md).

### Voice chat, speech input, and conversation mode

Voice features are now much more complete. AI Chat and Chat Interactions support **Text Only**, **Audio Input**, and **Conversation** modes, allowing speech-to-text input and text-to-speech output to work as part of a unified chat experience.

This release also adopts the `Microsoft.Extensions.AI` text-to-speech abstractions while preserving provider-driven voice discovery.

That means the platform is not only more powerful but also better aligned with the evolving .NET AI ecosystem. For users building conversational experiences, this brings Orchard Core much closer to the kind of modern, natural interaction patterns people increasingly expect.

See [AI Chat](../ai/chat.md), [AI Chat Interactions](../ai/chat-interactions.md), and [Artificial Intelligence Suite](../ai/index.md).

### Session analytics, resolution metrics, and conversion goals

The platform now includes significantly stronger chat analytics capabilities. This includes session metrics, AI-resolution-oriented analytics, and conversion goals so teams can measure outcomes such as leads, completed intents, and other business goals.

This matters because AI features become far more valuable when they are measurable. Orchard Core users can now think beyond "does the chat work?" and start asking "is the chat helping users, resolving tasks, and driving outcomes?"

See [AI Chat Analytics](../ai/chat-analytics.md).

### Data extraction and post-session processing

You can now extract structured data from chat sessions or run follow-up processing when a conversation ends. This supports scenarios such as lead capture, notifications, workflow triggers, or post-session integration logic.

This is a powerful bridge between conversational experiences and real business workflows. Instead of the chat ending when the user leaves, the session can become the trigger for downstream automation and operational processes.

See [AI Chat](../ai/chat.md) and [AI Chat with Workflows](../ai/workflows.md).

### File upload and document-aware sessions

Users can now upload files directly into a chat session and ask the AI to analyze, summarize, or answer questions about them. This is one of the major practical usability improvements in `2.0.0`.

It makes the AI experience feel much more grounded in the user's real work. Instead of forcing users to retype or restate everything, they can bring documents directly into the interaction and let the system help with analysis and understanding.

See [AI Documents](../ai/documents/index.md) and [AI Chat Interactions](../ai/chat-interactions.md).

### Admin widget and frontend widget support

In addition to frontend chat experiences, the platform now supports admin-facing widget scenarios so AI capabilities can be surfaced inside the Orchard Core admin environment as well.

This helps AI become part of the everyday Orchard authoring and administration experience, not just an isolated frontend feature. It makes the platform more useful to editors, operators, and administrators in addition to end users.

See [AI Chat](../ai/chat.md) and [Artificial Intelligence Suite](../ai/index.md).

### Centralized administration under Settings → Artificial Intelligence

One of the biggest usability improvements in this branch is that the AI platform is now much more configurable through the Orchard Core admin UI, especially under **Settings → Artificial Intelligence**. Deployments, defaults, interactions, analytics, and related options are now much easier to manage without custom code.

This is a big part of what makes the release so approachable: there is tremendous power here, but much more of it is now manageable through the product itself rather than hidden behind custom development work.

See [Artificial Intelligence Suite](../ai/index.md).

### Additional notable improvements

This branch also includes several important platform improvements that are worth calling out even though this page does not list every detailed change:

- improved initial prompt session start behavior
- refined chat session handler lifecycle behavior
- improved OpenXML Excel data extraction
- unified `DataSourceMetadata` modeling
- expanded recipe schema support for AI-assisted recipe authoring

Together, these improvements make the whole ecosystem feel more complete, more polished, and more ready for ambitious Orchard Core solutions.

See [AI Chat](../ai/chat.md), [AI Documents](../ai/documents/index.md), and [Recipes Module](../modules/recipes.md).

## Migration guidance

If you are upgrading from `v1.x`, treat this release as a major migration rather than a drop-in update. Start by reviewing the architecture and feature docs for the modules you use, then review the typed deployment migration guide before moving custom integrations forward.

The good news is that this migration effort unlocks a much bigger platform on the other side. `2.0.0` gives Orchard Core users access to far more modern AI patterns, cleaner deployment modeling, richer integrations, and a dramatically broader set of capabilities than the earlier generation of the project.
