# Unreleased

## Documentation

- Added a new **Framework** documentation section that documents each framework feature independently — covering core services, AI core, orchestration, chat, document processing, templates, tools, response handlers, context builders, SignalR, data storage, providers, data sources, MCP, and A2A.
- Each framework doc follows a consistent structure: quick start extension method, problem & solution, core interfaces, configuration, and implementation examples.
- Added an [MVC Example walkthrough](../framework/mvc-example.md) that documents all 13 sections of the `CrestApps.Mvc.Web` startup configuration.
- Updated the Introduction and Getting Started pages to present both Framework (standalone .NET) and Orchard Core paths.
- Updated the AI Suite index to reference framework docs for core concepts (orchestration, context builders, tools, etc.).
- Created `AddCrestAppsMcpClient()` and `AddCrestAppsMcpServer()` DI extensions that bundle previously scattered MCP service registrations into two cohesive calls.
- Cleaned up `CrestApps.Mvc.Web/Program.cs` with comprehensive section documentation aligned with the new framework docs.
- Documented why `AppDataConfigurationFileService` writes to `App_Data/appsettings.json`: it keeps admin-managed settings in the same reloadable configuration file so changes persist and flow back through ASP.NET Core configuration reload.
- Updated the framework docs to reflect the new `CrestApps.Infrastructure` layer, the `CrestApps.AI.Copilot` package name, and the renamed `CrestApps.Elasticsearch` / `CrestApps.Azure.AISearch` search-provider packages.
- Updated the framework docs again to reflect the new `CrestApps.Infrastructure.Abstractions` layer and the MVC sample's area-based feature layout.

## AI

- Restored shared framework-level A2A client support through a new `CrestApps.AI.A2A` project so Orchard Core and `CrestApps.Mvc.Web` now consume the same A2A connection models, authentication services, agent-card cache, tool registry provider, and discovery functions.
- Added MVC admin CRUD management for A2A host connections plus A2A capability selectors on AI profiles, AI profile templates, and chat interactions so stored connections flow into orchestration and remote A2A agents can be selected without duplicating MVC-only A2A service layers.
- Moved `SearchDocumentsTool`, `DataSourceSearchTool`, the default plain-text document reader, and the default AI document processor into the shared framework so Orchard Core and the MVC sample use the same host-agnostic document/data-source search pipeline.
- Added MVC JSON-backed settings for the default **AI Documents** index and default retrieval behavior, then updated MVC document uploads to process, embed, and index document chunks into that configured search index instead of only storing raw uploads.
- Moved the OpenXml and PDF ingestion readers into the shared framework so MVC and Orchard Core use the same document extraction path for embeddable uploads.
- Added MVC admin management for MCP connections, prompts, and resources, backed by shared framework MCP client/server services instead of MVC-specific protocol implementations.
- Added shared framework MCP FTP/SFTP resource metadata, handlers, and SSE/OAuth client plumbing so Orchard Core and MVC can expose the same MCP protocol capabilities with host-specific UI only.
- Updated the MVC admin experience with the requested dashboard section split, capability-oriented profile selection surfaces, persisted MCP server settings, and the A2A authentication form fix that reveals the correct fields per authentication mode.
- Replaced the MVC `ChatApiController` with three minimal API endpoints and aligned the sample app with the Orchard Core endpoint-style bootstrapping pattern.
- Moved chat interaction and chat session document upload/remove endpoints into the shared framework chat layer, added host-pluggable authorization and event hooks, and updated Orchard Core plus MVC to reuse the same HTTP contract.
- Updated the MVC chat UIs so chat interactions upload/remove knowledge documents immediately via AJAX, AI chat sessions expose session document uploads only when the profile enables them, AI Template list buttons match the AI Connections styling, and the MCP Hosts list no longer shows the redundant Details column.
- Aligned `CrestApps.Mvc.Web` Copilot settings and Copilot orchestrator readiness checks with Orchard Core so the sample app now respects the `NotConfigured` state, hides the wrong auth-specific UI when Copilot is disabled, and only marks Copilot as ready when the same OAuth/BYOK requirements are satisfied.
- Finished the remaining MVC Copilot parity gaps by wiring Orchard-style Copilot warnings and GitHub sign-in/status UI into AI profile templates, copying template Copilot metadata into created profiles, renaming the AI Deployment editor to use `Technical name` and `Model name`, and persisting chat interaction deletes so confirmed removals actually disappear.
- Fixed the MVC Copilot orchestrator panel so it correctly shows the "not configured" warning when a user switches to the Copilot orchestrator, regardless of the current authentication type. Previously the panel toggle JavaScript was only rendered for GitHubOAuth mode, so the warning never appeared when Copilot was unconfigured.
- Fixed the MVC Chat Interaction chat sidebar orchestrator selector so the Copilot panel toggle JavaScript can find it (added the required `name` attribute).
- Simplified MVC Chat Interaction creation flow: clicking "New Chat" now auto-creates a blank interaction and immediately opens the full chat view with settings sidebar and chat window side by side, eliminating the separate two-step create-then-chat workflow.
- Registered `AddLocalization()` in the MVC startup so `IStringLocalizerFactory` is available to shared framework endpoints (fixes a `NullReferenceException` in the document upload endpoints).
- Added a complete **Articles** admin feature to the MVC sample app demonstrating custom knowledge-base data sources: CRUD management with Markdown descriptions, YesSql-backed catalog storage with `ArticleIndex`, automatic search indexing via `ArticleIndexingHandler` (Elasticsearch and Azure AI Search), configurable pagination settings (default 25 items per page) shared across all admin lists, a reusable `_Pager` partial, six seed articles covering AI and search indexing topics, and a registry-backed index-profile source system so `"Articles"` appears as a first-class index profile type for both Elasticsearch and Azure AI Search.
- Split the framework search-provider integrations into dedicated `CrestApps.Elasticsearch` and `CrestApps.Azure.AISearch` projects so provider-specific dependencies and services (client setup, index managers, document managers, vector search, OData translation, and source readers/content managers) now live in separate reusable libraries. The older `CrestApps.AI.DataSources.*` framework projects were removed so the provider libraries are the canonical indexing backends for data sources, documents, and memory.
- Added a lower `CrestApps.Infrastructure` framework layer for host-agnostic helpers and constants, moved prompting into the core framework folder, and renamed the Copilot integration project to `CrestApps.AI.Copilot` so the framework structure reads as infrastructure first and AI on top.
- Added `CrestApps.Infrastructure.Abstractions` and moved search, indexing, document-chunk, and data-source contracts out of `CrestApps.AI.Abstractions` so `CrestApps.Azure.AISearch` and `CrestApps.Elasticsearch` now depend on infrastructure-level contracts instead of AI-level abstractions.
- Reorganized the remaining `CrestApps.AI.Abstractions` project into feature folders and split the MVC sample's former catch-all admin surface into dedicated `AI`, `AIChat`, `ChatInteractions`, `A2A`, `Mcp`, `DataSources`, and `Indexing` areas while keeping shared settings and articles under `Admin`.
- Consolidated MVC admin-managed settings into a single `App_Data/appsettings.json` override file layered on top of `appsettings.json` and `appsettings.Development.json`. Settings now live only as nested sections in those three files; the legacy per-setting JSON files are no longer part of the configuration flow. Search providers now bind from `CrestApps:Elasticsearch` and `CrestApps:AzureAISearch`, matching the existing local override file structure.
- Removed MVC-only backward-compatibility shims for the old `AddElasticsearchIndexingSource()` and `AddAzureAISearchIndexingSource()` registration names so the sample app exposes only the clean `AddElasticsearchDataSource()` and `AddAzureAISearchDataSource()` API surface.
- Refined the MVC index-profile registration surface so built-in provider/type combinations now use explicit extensions such as `AddElasticsearchAIDocumentSource()`, `AddElasticsearchAIDataSource()`, and `AddAzureAISearchAIMemorySource()`, which register both the source descriptor and the matching `IIndexProfileHandler`.
- Updated the MVC shared layout to always show server-side `ModelState` errors and redirected `ErrorMessage` alerts, and relaxed MVC index-profile deletion so admins can still remove a local profile when the remote Elasticsearch or Azure AI Search index was already deleted manually or the stored profile can no longer resolve a remote index name.
- Relaxed MVC index-profile deletion further so admins can still remove a local profile even when the original provider registration is no longer present in the app.
- Fixed the shared `CatalogManager<T>.DeleteAsync()` path to commit catalog deletions through `SaveChangesAsync()`, which also fixes MVC index-profile deletes persisting to the YesSql store.
- Updated the MVC chat UIs so Chat Interactions no longer shows a redundant attachment button next to the message box, the widget now exposes direct session-document uploads whenever the selected AI profile enables them, and MVC index profiles now require embeddings for AI Documents / AI Memory / Data Source types, hide the selector for Articles, compose remote index names from provider `IndexPrefix` + index name, provision the remote index before saving the local profile, reject creates when the remote index already exists, lock index/provider/type/embedding after creation, and delete the remote search index when the profile is removed.
- Refined the MVC index-profile implementation to align even more closely with Orchard Core indexing: the sample now resolves provider-specific work through keyed `ISearchIndexManager`, those methods accept `IIndexProfileInfo` instead of raw index-name strings, and shared `IIndexProfileHandler` implementations now validate embedding requirements and provide schema fields so the temporary `ISearchIndexProfileManager` layer is no longer needed.
- Fixed MVC Chat Interactions data-source RAG by persisting `DataSourceMetadata` and `AIDataSourceRagMetadata` through the shared chat settings pipeline, seeding default RAG metadata when a chat interaction is created with a data source, and moving both the chat-settings handler and data-source preemptive RAG handler registrations into the shared framework/provider extensions so knowledge-base context is injected during chat requests without MVC-only wiring.
- Fixed MVC Chat Interaction settings persistence so the shared save path now applies `DocumentTopN` and `AgentNames`, stores prompt-template selections in `PromptTemplateMetadata`, and reloads those saved prompt templates back into the chat editor after refresh.
- Fixed MVC Chat Interaction data-source reload so the chat editor now rehydrates the saved selection from `DataSourceMetadata` and the saved RAG values from `AIDataSourceRagMetadata` when the page is reopened.
- Expanded the MVC Knowledge tabs so Chat Interactions and AI Profiles both expose the `AIDataSourceRagMetadata` fields (`Strictness`, `TopNDocuments`, `IsInScope`, and `Filter`), the Chat Interaction knowledge tab now separates data source, RAG, and document-upload sections visually, and chat document uploads now show visible progress, per-file status, and success/error feedback instead of only logging failures to the console.
- Aligned the MVC AI Profile editor with the Orchard Core storage model so `AIProfileSettings`, data extraction, post-session processing, and profile memory use the `Settings` store APIs while profile documents and session documents continue to use metadata in `Properties`. The MVC profile mapper now also uses `Alter<>` consistently for metadata-backed writes, including prompt templates, analytics, documents, session documents, and Copilot metadata, and fixes the saved RAG filter/top-N mapping.
- Hardened the MVC sample's view models so every `SelectListItem` option collection now uses `IEnumerable<SelectListItem>` with `[BindNever]`, and aligned chat-interaction creation plus live settings saves on `Alter<>` so prompt templates, Copilot metadata, and knowledge-base metadata persist consistently.
- Fixed flattened `JsonExtensionData` reloads for AI profiles and other extensible entities by teaching the shared `As<T>()` / `TryGet<T>()` helpers to deserialize dictionary-backed property values in addition to `JsonElement` and `JsonNode`, which restores saved data-source selections after MVC profile edits are reloaded from storage.
- Fixed AI profile JSON serialization so `Settings` stays as the nested settings bag while extensible metadata remains flattened in `Properties`, avoiding the `JsonExtensionData` conflict that occurs when `System.Text.Json` sees more than one extension-data member on the same profile model.
- Finished the remaining profile-side storage cleanup so shared completion-context building, agent profile inspection/listing tools, and the chat-session close background task now read `...Metadata` from `Properties` and `...Settings` from the `Settings` bag consistently with the Orchard Core implementation.
- Updated the MVC Chat Interaction editor so model-tuning parameters stack one-per-row until extra-large layouts, each parameter label shows the same hover help affordance used in Orchard Core, and duplicate chat document uploads are rejected with user-visible feedback instead of being processed twice.
- Reorganized the Orchard Core AI Profile editor so the profile-level data source selector, RAG controls (`Strictness`, retrieved documents, in-scope answers only, and filter), profile documents, and session-document uploads now all live under a single **Knowledge** tab instead of splitting data source settings across the General, Parameters, and Documents sections.
- Completed the remaining parameter-help consistency work so MVC AI Profiles, MVC profile-source AI Templates, Orchard Core AI Profiles, and Orchard Core profile-source AI Templates now all show the same `(?)` tooltip help on model parameters as Chat Interactions.
- Fixed the MVC **AI Templates** navigation links so the menu and dashboard now route to the actual `AITemplateController` instead of the obsolete `Template` controller name that returned 404.
- Registered `IAIProfileStore` in the Orchard Core AI core service extension so migrations and other Orchard-hosted profile services can resolve the shared profile store through DI consistently with the framework host.
- Fixed the remaining Orchard Core chat-interaction editor cleanup by moving data source + RAG settings and document retrieval settings into the shared **Knowledge** tab, renaming the document retrieval label to **Document Top N**, and registering `DefaultSpeechVoicePresenter` in Orchard Core AI core services so the profile chat-mode drivers resolve cleanly.
- Added MVC AI settings parity for AI Memory, data-source defaults, default orchestrator preemptive RAG, and chat-interaction memory defaults; completed the missing Knowledge-tab tooltip help in the MVC chat UI; registered the missing MVC deployment manager needed by the settings page; and hardened the PR #367 CodeQL paths around logging, file-path validation, and MCP transport/auth normalization.
- Extended the Orchard Core legacy AI document-type migration so existing tenants rerun the Orchard-to-framework type rewrite and recover legacy `AIProfile`, `AIChatSession`, and `AIChatSessionPrompt` rows that would otherwise fail YesSql type resolution.
- Renamed the reusable prompting layer to `CrestApps.Templates`, updated registration to `AddTemplating()` / `AddTemplatesFromAssembly()`, and switched file-system plus embedded-resource discovery from `AITemplates/...` to `Templates/...`.
- Added a new `AddAITemplating()` helper in `CrestApps.AI` so AI hosts can register the shared templating engine plus the built-in `SystemPrompt` and `Profile` template sources with a single call.
- Continued the MVC feature reorganization by relocating feature-specific hubs, minimal API endpoints, background services, indexes, and the `_ChatWidget` partial into their matching area folders.
- Moved the shared MVC `_CopilotConfig` partial into `Views/Shared` so AI and ChatInteractions area views resolve the Copilot configuration panel through the standard Razor shared-view lookup path instead of failing at runtime outside the Admin area.
- Removed the legacy CrestApps-owned text-to-speech abstraction surface so AI client factories and providers now use `Microsoft.Extensions.AI.ITextToSpeechClient` and the built-in `TextToSpeech*` models directly.
- Fixed the remaining AI admin consistency gaps across Orchard Core and `CrestApps.Mvc.Web`: MVC settings now reload the just-saved values on the first redirect, MVC AI Profiles and Profile-source AI Templates now align on **Instructions** / **Knowledge** organization (including prompt templates, user memory, and RAG fields), Chat Interactions place **Knowledge** before **Capabilities**, the AI Templates list keeps action buttons on one line, the admin chat widget only renders when a profile is selected in site settings, and Orchard Core AI Core now registers `ISpeechVoiceResolver` so the AI Profile editor resolves speech voice presenters correctly.

- Created `CrestApps.Mvc.Samples.McpClient` sample project demonstrating MCP client connectivity to `CrestApps.Mvc.Web`, with tools, prompts, resources, and resource template browsing.
