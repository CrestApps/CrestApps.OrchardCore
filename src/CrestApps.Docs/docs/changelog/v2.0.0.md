# Version 2.0.0 Release Notes

Version `2.0.0` is the major upgrade from the `v1.x` line to the current `v2` platform. It moves CrestApps.OrchardCore to Orchard Core 3 on .NET 10, expands the solution into a much broader modular platform, and reshapes how AI deployments, chat experiences, retrieval, external integrations, and omnichannel communication are modeled.

This page is the consolidated release guide for `v1.x` to `v2.0.0`. It combines breaking changes, new features, and migration guidance in one place.

## Before you upgrade

Before upgrading from `v1.x` to `v2.0.0`, back up every tenant database and review your custom code, package references, enabled features, and configuration.

The upgrade is not only a package bump. It includes:

- a platform upgrade to `.NET 10`
- an Orchard Core upgrade to the `3.0.0-preview` line used by this repo
- a move to shared `CrestApps.Core` packages
- a typed AI deployment model
- feature and module boundary changes across AI, retrieval, and provider integrations

At a high level, the platform changes are:

| Area | v1.x | v2.0.0 |
| --- | --- | --- |
| .NET SDK | `8.0.100` | `10.0.100` |
| Target frameworks | `net8.0;net9.0` | `net10.0` |
| Orchard Core | `2.1.x` line | `3.0.0-preview` line |
| NuGet feeds | NuGet only | NuGet + Orchard Core preview + CrestApps Core preview |
| Shared code model | Mostly repo-local core projects | Shared `CrestApps.Core` packages plus Orchard integration modules |
| AI deployments | Embedded provider and deployment selectors | First-class typed AI deployments |
| Retrieval architecture | Mostly Azure OpenAI BYOD-centric | Provider-agnostic Data Sources, Documents, and Memory modules |
| Documentation | README-driven | Docusaurus documentation under `src\CrestApps.Docs` |

## Breaking changes

### Upgrade to .NET 10 and Orchard Core 3

`v2.0.0` requires `.NET 10` and the Orchard Core `3.0.0-preview` package line used by this repository.

**What you need to do**

1. Install the `.NET 10` SDK.
2. Move application projects to `net10.0`.
3. Update package references to the `v2` package line.
4. Build and resolve any compile-time issues in custom modules before deploying.

### Shared framework extraction to `CrestApps.Core`

Large parts of the reusable AI infrastructure are no longer implemented only inside Orchard-facing projects. The `v2` line depends on shared `CrestApps.Core` packages, while Orchard modules provide integration, UI, and runtime wiring.

**What changed**

- several Orchard-only implementation details moved into shared `CrestApps.Core.*` packages
- package references and namespace usage changed for custom code that depended on repo-local v1 internals
- samples and startup wiring follow the new package layout

**What you need to do**

1. Review custom project references.
2. Replace direct references to moved v1 implementation assemblies with the matching `CrestApps.Core` packages and namespaces.
3. Rebuild any custom modules against the new dependency graph.

### Typed AI deployments replace legacy deployment selectors

This is one of the most important breaking changes in the upgrade.

In `v1.x`, many AI settings stored provider and deployment information inline or by name. In `v2.0.0`, deployments are first-class records and related models now use typed deployment selectors such as `ChatDeploymentId` and `UtilityDeploymentId`.

**What changed**

- AI deployments are now managed as their own records
- profiles, chat interactions, and templates resolve deployments through the typed deployment model
- deployment defaults are managed centrally through site settings

**What you need to do**

1. Update recipes, appsettings, and custom code that read or write legacy deployment-name fields.
2. Review the [Migrating to Typed AI Deployments](../ai/migration-typed-deployments.md) guide.
3. After upgrade, verify that site settings contain the expected default chat deployment and other default deployment selectors.

#### Deployment connection picker and legacy alias cleanup

- the deployment editor now loads connection choices from the provider connection catalog instead of the merged `AIProviderOptions` snapshot
- provider-specific connection pickers now show the stored connections that belong to the selected provider
- deployment documents now persist the provider connection `Name` in `ConnectionName` instead of the connection record `ItemId`, and a follow-up migration normalizes older stored values
- the legacy `ConnectionNameAlias` deployment property is no longer written or used at runtime
- the deployment index backfill now also infers missing deployment types from legacy provider-connection deployment names and legacy `AIProfile` selectors such as `DeploymentId`, `ChatDeploymentId`, and `UtilityDeploymentId`, so older dictionary documents remain importable

#### AI chat citation rendering and chat UI asset refresh

- the AI chat admin UI, admin widget, frontend widget, and chat interaction surfaces now render `[doc:N]` markers as superscript citations with a linked reference list when the underlying references can be resolved
- the OrchardCore chat modules now use `@crestapps/ai-chat-ui` `1.0.0-preview.20`
- the resource manifests were updated to the matching preview.20 CDN URLs and `sha384` integrity hashes
- the frontend chat widget now clears theme-added paragraph bottom spacing inside rendered messages so single wrapped `<p>` blocks do not create extra gaps
- the shared `list-management-ui` script now registers `window.listManagementUI` explicitly so admin catalog pages such as **Chat Interactions** can initialize search and bulk actions reliably

### AI feature boundaries changed

Several v1 capabilities still exist in `v2`, but not under the same module or feature boundaries.

| v1 capability | v1 location | v2 location or action |
| --- | --- | --- |
| Base AI deployments feature | `CrestApps.OrchardCore.AI.Deployments` | Deployment management is part of the broader AI platform and typed deployment model |
| Base AI tools feature | `CrestApps.OrchardCore.AI.Tools` | Tooling remains part of the orchestrated AI platform; review [AI Tool Management](../ai/tools.md) |
| Azure OpenAI BYOD and provider-bound data sources | `CrestApps.OrchardCore.OpenAI.Azure.*` | Move to [AI Data Sources](../ai/data-sources/index.md), [AI Documents](../ai/documents/index.md), and [AI Memory](../ai/memory.md) |
| Azure AI Search data-source feature | `CrestApps.OrchardCore.OpenAI.Azure.AISearch` | Use `CrestApps.OrchardCore.AI.DataSources.AzureAI` and related document and memory modules |
| Elasticsearch data-source feature | `CrestApps.OrchardCore.OpenAI.Azure.Elasticsearch` | Use `CrestApps.OrchardCore.AI.DataSources.Elasticsearch` and related document and memory modules |
| MongoDB data-source feature | `CrestApps.OrchardCore.OpenAI.Azure.MongoDB` | No direct shipped `v2` replacement was found; plan a custom replacement before upgrade |
| Simple chat-only experience | `CrestApps.OrchardCore.AI.Chat` | v2 keeps chat and adds Chat Interactions, Copilot, analytics, admin widget, and response handlers |
| MCP integration | `CrestApps.OrchardCore.AI.Mcp` | Still present, but expanded with client and server support plus FTP and SFTP MCP resource modules |

**What you need to do**

1. Map each enabled v1 feature to its v2 module and feature equivalent.
2. Re-enable any new optional modules explicitly.
3. Do not assume that old feature IDs or old provider-specific boundaries still exist unchanged.

### Azure OpenAI retrieval architecture changed

In `v1.x`, bring-your-own-data scenarios were heavily tied to Azure OpenAI provider features. In `v2.0.0`, retrieval is modeled as a broader provider-agnostic platform with dedicated Data Sources, Documents, and Memory modules.

**What changed**

- Azure OpenAI no longer owns the full retrieval story
- retrieval, indexing, and memory capabilities moved into dedicated AI modules
- document parsing and storage are now modularized

**What you need to do**

1. Review old Azure OpenAI data-source configurations.
2. Re-map those scenarios to the new Data Sources, Documents, and Memory modules.
3. If you relied on MongoDB-backed data sources in v1, plan a replacement before upgrade.

### Package feeds and restore setup changed

The `v2` line uses preview feeds for Orchard Core and CrestApps Core packages.

**What you need to do**

1. Adopt the v2 `NuGet.config`.
2. Make sure the Orchard Core preview and CrestApps Core preview feeds are available in every build environment.
3. Do not keep the old v1 feed setup and expect restore to succeed.

### AI administration and navigation changed

AI administration is reorganized in `v2.0.0`.

**What changed**

- settings and management screens are grouped differently
- profile and chat-interaction editors are more source-agnostic
- deployment defaults are more centralized
- documentation moved to the Docusaurus site

**What you need to do**

Review the current admin navigation after upgrade and update any operator documentation or onboarding instructions that referenced the old v1 admin flow.

## New features

### Orchestrator-based AI execution

The AI platform moves away from a simpler prompt-routing model and toward an orchestrator-oriented execution model that can coordinate tools, retrieval, templates, agents, and external integrations.

This is the architectural foundation for the broader v2 AI platform.

See [Artificial Intelligence Suite](../ai/index.md), [AI Agent](../ai/agent.md), and [AI Tool Management](../ai/tools.md).

### AI Chat Interactions

`v2.0.0` adds ad-hoc chat interactions that do not require a predefined AI profile for every scenario.

This enables:

- session-specific chat behavior
- more dynamic admin and frontend AI experiences
- richer file, image, and interaction-driven workflows

See [AI Chat Interactions](../ai/chat-interactions.md).

### AI chat session initialization fix

The admin AI chat session page now keeps the placeholder element rendered even when a historical session already has messages. This prevents the shared chat UI runtime from aborting initialization when users reopen an existing session from the history list.

### Orchard time abstraction alignment

The remaining Orchard-facing AI services, hubs, handlers, and background tasks now use `OrchardCore.Modules.IClock` consistently instead of `TimeProvider`, keeping timestamp creation aligned with the rest of the multi-tenant application code.

### AI profile handler deduplication

The Orchard integration now defers to the shared `CrestApps.Core.AI.Handlers.AIProfileHandler` implementation instead of keeping duplicate local AI profile handlers for prompt-template and deployment validation logic.

### Frontend asset rebuild stability

The repository `gulpfile.js` now waits for each CSS and JavaScript output stream to finish before reporting asset task completion. This keeps `npm run rebuild` stable on current Node.js and Gulp versions and avoids the premature stream-close failure that could stop local frontend builds.

### AI Memory profile cleanup confirmation flow

The user profile AI Memory editor now keeps the **Danger zone** warning but replaces the inline clear-memory confirmation checkboxes with a dedicated destructive action button that opens a separate confirmation page before deletion is processed.

### AI data source index backfill migration

The AI Data Sources module now adds a dedicated `AIDataSourceIndex` schema migration and backfills legacy `DictionaryDocument<AIDataSource>` records from the Orchard `Document` table into the current indexed data-source store structure, including older `CrestApps.AI` and `CrestApps.Core.AI` document type names. During import, the migration also copies a legacy `ProfileSource` value into `SourceIndexProfileName` when the newer field is missing and uses `IAIDataSourceStore.GetAllAsync()` for existing-record detection so the migration stays inside the configured store pipeline even when the map index starts empty.

The follow-up metadata migration now also infers missing source-field mappings from the selected source index profile, assigns an existing provider-matched knowledge-base index when available, and can create the default **AI Knowledge Base Warehouse** profile on demand so legacy data sources no longer fail import just because newer required index fields were absent from the original document payload.

The index-profile embedding metadata normalization step now rewrites stored embedding selectors to the deployment record `Name` instead of leaving model names such as `text-embedding-3-small` in place. This keeps knowledge-base index profiles aligned with the unique technical deployment name that runtime indexing resolves through `IAIDeploymentManager.FindByNameAsync(...)`.

The same normalization pass now reruns for tenants already migrated to the earlier version, accepts legacy Azure provider aliases such as `Azure` and `AzureOpenAIOwnData` when matching current Azure OpenAI deployments, and strips historical `Azure-` selector prefixes before rewriting the selector to the canonical deployment record name.

The follow-up rerun also covers tenants where the index-profile normalization executed before the legacy AI deployment import completed during startup. In that case, runtime indexing now repairs a unique embedding model selector such as `text-embedding-3-small` to the imported deployment record name, for example `Azure-text-embedding-3-small`, and persists the corrected selector back to the index profile.

### Security and webhook hardening updates

- MCP API key authentication now uses constant-time secret comparison instead of ordinal string equality.
- A2A and MCP host authentication now default to `OpenId` when no authentication type is configured, so anonymous access must be chosen explicitly.
- Azure Event Grid webhook authentication now supports metadata-backed bearer token validation through `AADMetadataAddress`, uses constant-time SAS key comparison, and documents the exact webhook endpoint and required configuration.
- The Twilio webhook now reuses the parsed form payload instead of reading the request body twice.
- AI data source name filtering now correctly matches records whose display text contains the requested search term.

### Npm audit dependency remediation

The root npm workspace now pins patched transitive versions of `lodash`, `yaml`, and `brace-expansion` through `overrides`. This removes the `npm install` audit warnings that were coming from the shared asset-build toolchain.

### Preview package feeds

The current checked-in `NuGet.config` restores `CrestApps.Core*` packages from the shared CrestApps Core preview feed and restores Orchard Core packages from the Orchard Core preview feed. Build environments must have access to both preview sources for restore and build to succeed.

### Documentation alignment updates

The Orchard Core docs were refreshed so setup, upgrade, and module usage guidance match the current codebase more closely.

This included:

- removing outdated repo-local `crestapps-core-local` feed instructions from the setup guide
- linking the getting-started guide directly to the current upgrade and typed-deployment migration documentation
- correcting MCP client feature IDs and admin navigation labels
- refreshing provider setup wording and the docs contributor guidance

### Orchard data-source sync background resolution fix

The Orchard Core data-source module now explicitly overrides `IAIDataSourceIndexingService` with the Orchard-specific implementation before queued sync work is handed to `HttpBackgroundJob.ExecuteAfterEndOfRequestAsync(...)`.

This keeps index-profile rebuilds and queued data-source synchronization on the Orchard adapter path instead of falling back to the shared core registration that expects non-Orchard indexing services.

### Orchard source-index synchronization fan-out fix

Synchronizing a source index profile such as Orchard's built-in **Content** index now queues a full re-sync for every AI data source mapped to that source profile.

The Orchard data-source indexing service also now resolves the configured AI knowledge-base profile by profile `Name` consistently during full re-sync work, so rebuilding a source index correctly repopulates the mapped `ai_knowledge_base_warehouse` index instead of silently skipping the data source.

### AI index editor guardrails

The AI Memory index-profile editor now treats the embedding deployment as a creation-time choice only. Once the index profile already exists, the deployment is shown as read-only and the display driver no longer attempts legacy fallback behavior because the migration now normalizes existing records in advance.

The AI Data Sources editor also now filters the **Source Index** selector to exclude AI-managed index types such as **AI Documents**, **AI Memory**, and **Data Source** knowledge-base indexes, leaving only valid source indexes available for new data sources.

### Centralized package-tag metadata

Package tags are now computed centrally for the full solution during packing instead of being maintained inconsistently across individual project files. The generated tags now include host, project-type, and feature keywords such as `Module`, `Core`, `AI`, `Chat`, `MCP`, `OpenAI`, `Omnichannel`, `Recipes`, and provider-specific terms, making the resulting NuGet metadata more consistent and easier to search.

### Copilot chat experience

The new Copilot integration introduces a GitHub Copilot chat experience for Orchard Core.

This brings a more modern AI assistant interface into the platform while still allowing Orchard-specific tools, content access, and orchestration behavior.

The Copilot site settings now also include a **Not configured** authentication mode so administrators can disable the orchestrator without deleting previously stored credentials. GitHub OAuth client ID and client secret validation only runs when that authentication mode is selected, and an existing stored client secret remains valid until an administrator explicitly replaces it.

See [AI Chat Copilot Integration](../ai/copilot.md).

The **Settings → Artificial Intelligence** page now keeps the chat-interaction memory guidance inside the **Memory** section context instead of telling administrators to navigate away from the page they are already on, and the AI settings cards use a stable order so **Admin Widget**, **Copilot**, and **Claude** render back to back in the expected sequence.

### Claude integration

`v2.0.0` adds Claude support as a first-class Orchard integration.

This includes provider-specific settings and Claude-oriented deployment and model selection support in the AI platform.

See [Claude](../ai/claude.md).

### Prompt templates and profile templates

Templates become first-class reusable building blocks in `v2.0.0`.

This includes:

- reusable prompt templates
- reusable profile templates
- more consistent AI experience definitions across tenants and use cases

See [AI Prompt Templates](../ai/prompt-templates.md) and [AI Profile Templates](../ai/profile-templates.md).

### AI Data Sources

Data Sources are now a first-class provider-agnostic capability instead of being tied mainly to Azure OpenAI BYOD scenarios.

This allows teams to design retrieval and external data strategies independently of a single provider story.

The Orchard Core integration now also bridges `IDocumentIndexHandler` source-index notifications into an Orchard-managed `IAIDataSourceIndexingQueue`, which batches request-scoped work and dispatches it with `HttpBackgroundJob.ExecuteAfterEndOfRequestAsync(...)` so Elasticsearch and Azure AI Search document writes automatically refresh mapped knowledge-base indexes without waiting for a timer. The nightly alignment worker still repairs drift asynchronously.

See [AI Data Sources](../ai/data-sources/index.md).

### AI Documents

`v2.0.0` adds a dedicated document pipeline for upload, parsing, chunking, indexing, and retrieval.

This includes support for:

- PDF documents
- OpenXML documents
- dedicated document storage and indexing integrations
- document-aware AI workflows and chat experiences

See [AI Documents](../ai/documents/index.md), [PDF support](../ai/documents/pdf.md), [OpenXML support](../ai/documents/openxml.md), and [Azure Blob Storage](../ai/documents/azure-blob-storage.md).

### AI Memory

`v2.0.0` adds persistent AI memory support so assistant behavior can retain context across sessions and use provider-backed retrieval or indexing strategies.

See [AI Memory](../ai/memory.md).

### Response handlers and live-agent handoff

Chat handling is now extensible through pluggable response handlers.

This enables:

- AI-driven responses
- deferred external processing
- live-agent handoff
- hybrid AI and human support scenarios

See [Response Handlers](../ai/response-handlers.md) and [Chat UI Notifications](../ai/chat-notifications.md).

### MCP client and server support

The platform now supports both exposing Orchard Core as an MCP server and consuming external MCP servers as a client.

`v2.0.0` also expands the MCP surface with FTP and SFTP resource modules.

See [Model Context Protocol](../ai/mcp/index.md), [MCP Server](../ai/mcp/server.md), [MCP Client Integration](../ai/mcp/client.md), [FTP MCP resources](../ai/mcp/ftp.md), and [SFTP MCP resources](../ai/mcp/sftp.md).

### Agent-to-Agent support

`v2.0.0` adds Agent-to-Agent support so Orchard-hosted AI solutions can participate in broader multi-agent workflows.

See [A2A](../ai/a2a/index.md).

### AI chat analytics and richer chat UX

The v2 line also adds richer AI chat capabilities, including:

- session analytics
- improved admin and frontend widget scenarios
- more deployment-aware and profile-aware chat flows
- the **Prompt subject** editor field only appears for **Template generated prompt** profiles and profile-source templates
- stronger post-session processing support

See [AI Chat](../ai/chat.md) and [AI Chat Analytics](../ai/chat-analytics.md).

### Omnichannel Communications

`v2.0.0` introduces the Omnichannel platform and its related modules.

This adds:

- a unified omnichannel foundation
- management features
- Event Grid integration
- SMS channel support

See [Omnichannel Communications](../omnichannel/index.md), [Management](../omnichannel/management.md), [Event Grid](../omnichannel/event-grid.md), and [SMS](../omnichannel/sms.md).

### Recipes and expanded samples

The v2 solution also grows the surrounding platform surface with:

- expanded recipe support
- additional sample applications
- a dedicated documentation project

## Migration from v1.x

### Step 1: Back up tenant data

Back up every tenant database before first startup on `v2.0.0`.

This is especially important for tenants that already contain:

- AI provider connections
- AI deployments
- AI profiles
- AI chat sessions

### Step 2: Update runtime, packages, and feeds

1. Install `.NET 10`.
2. Update to the `v2.0.0` package line.
3. Move projects to `net10.0`.
4. Adopt the v2 `NuGet.config` and preview package feeds.

### Step 3: Re-map enabled features

Review the v1 modules and provider-specific features you enabled and map them to the current v2 module set.

The major v2 module additions include:

- `CrestApps.OrchardCore.AI.A2A`
- `CrestApps.OrchardCore.AI.Chat.Claude`
- `CrestApps.OrchardCore.AI.Chat.Copilot`
- `CrestApps.OrchardCore.AI.Chat.Interactions`
- `CrestApps.OrchardCore.AI.DataSources`
- `CrestApps.OrchardCore.AI.DataSources.AzureAI`
- `CrestApps.OrchardCore.AI.DataSources.Elasticsearch`
- `CrestApps.OrchardCore.AI.Documents`
- `CrestApps.OrchardCore.AI.Documents.Azure`
- `CrestApps.OrchardCore.AI.Documents.AzureAI`
- `CrestApps.OrchardCore.AI.Documents.Elasticsearch`
- `CrestApps.OrchardCore.AI.Documents.OpenXml`
- `CrestApps.OrchardCore.AI.Documents.Pdf`
- `CrestApps.OrchardCore.AI.Mcp.Resources.Ftp`
- `CrestApps.OrchardCore.AI.Mcp.Resources.Sftp`
- `CrestApps.OrchardCore.AI.Memory`
- `CrestApps.OrchardCore.AI.Memory.AzureAI`
- `CrestApps.OrchardCore.AI.Memory.Elasticsearch`
- `CrestApps.OrchardCore.AI.Prompting`
- `CrestApps.OrchardCore.Omnichannel`
- `CrestApps.OrchardCore.Omnichannel.EventGrid`
- `CrestApps.OrchardCore.Omnichannel.Managements`
- `CrestApps.OrchardCore.Omnichannel.Sms`
- `CrestApps.OrchardCore.Recipes`

### Step 4: Migrate AI deployments and defaults

The v2 upgrade path must preserve legacy AI connections, deployments, profiles, and chat-session data from real `v1.x` tenant databases.

During deployment migration, make sure you verify:

- legacy `DictionaryDocument<AIDeployment>` records were imported
- missing v1 deployment fields such as `ModelName`, `ClientName`, and `Type` were normalized correctly
- default deployment site settings were backfilled
- site settings contain at least a default chat deployment when chat deployments exist

The default deployment backfill order should be verified as:

1. UI-configured default deployment
2. legacy configured default from existing settings or appsettings-backed configuration
3. the first available deployment of that type

For the deployment model details, see [Migrating to Typed AI Deployments](../ai/migration-typed-deployments.md).

### Step 5: Review custom code

Review custom modules for:

- moved namespaces and package references
- legacy deployment-name usage
- older feature IDs
- provider-specific assumptions that no longer match the v2 module architecture

### Step 6: Validate the upgraded tenant

After startup on `v2.0.0`, validate:

1. AI provider connections are present.
2. AI deployments are present and usable.
3. AI profiles load correctly.
4. AI chat sessions remain available.
5. Default chat deployment and related site defaults are set correctly.
6. Any enabled retrieval, document, or provider integrations still map to the intended modules.
