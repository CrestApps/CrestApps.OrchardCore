# Version 2.0.0 Release Notes

Version `2.0.0` is the major upgrade from the `v1.x` line to the current `v2` platform. It moves CrestApps.OrchardCore to Orchard Core 3 on .NET 10, expands the solution into a much broader modular platform, and reshapes how AI deployments, chat experiences, retrieval, external integrations, and omnichannel communication are modeled.

This page is the consolidated release guide for `v1.x` to `v2.0.0`. It combines breaking changes, new features, and migration guidance in one place.

## Highlights in 2.0.0

- **Platform upgrade** to `.NET 10` and the Orchard Core `3.0.0-preview` line used by this repository
- **Expanded AI platform** with deployment records, provider connections, orchestrators, tools, data sources, documents, memory, MCP, and A2A features
- **Cleaner AI configuration model** built around current connection and deployment records
- **Broader Orchard module surface** including Omnichannel Management, Time Zones, Content Fields, DNC Registry, SignalR, Roles, Content Transfer, and more
- **Improved recipe and schema coverage** for Orchard steps, content payloads, settings, and admin provisioning
- **Smarter AI recipe provider enums** so `AIProviderConnections.Source` and `ClientName`, plus `AIDeployment.ClientName`, now enumerate the provider names currently registered for the tenant
- **Canonical Docusaurus documentation** under `src\CrestApps.Docs` for the current platform structure

## Recommended migration path

Use this order when moving a tenant or solution from `v1.x` to `v2.0.0`:

1. Back up every tenant database and export any important recipes or deployment plans.
2. Install the `.NET 10` SDK and move application projects to `net10.0`.
3. Update package feeds and package references to the `v2.0.0` line.
4. Review custom code for moved namespaces, shared `CrestApps.Core` packages, and updated async interface signatures.
5. Start the application once and allow the data migrations to complete.
6. Review AI provider connections, AI deployments, default deployment settings, AI profiles, and chat data in the admin UI.
7. Rebuild and validate tenant features such as AI chat, data sources, documents, MCP, Omnichannel imports, and custom modules.

## Before you upgrade

Before upgrading from `v1.x` to `v2.0.0`, back up every tenant database and review your custom code, package references, enabled features, and configuration.

The upgrade is not only a package bump. It includes:

- a platform upgrade to `.NET 10`
- an Orchard Core upgrade to the `3.0.0-preview` line used by this repo
- a move to shared `CrestApps.Core` packages
- a typed AI deployment model
- feature and module boundary changes across AI, retrieval, and provider integrations

At a high level, the platform changes are:

| Area | v1.x | v2.0.0 |
| --- | --- | --- |
| .NET SDK | `8.0.100` | `10.0.100` |
| Target frameworks | `net8.0;net9.0` | `net10.0` |
| Orchard Core | `2.1.x` line | `3.0.0-preview` line |
| NuGet feeds | NuGet only | NuGet + Orchard Core preview + CrestApps Core preview |
| Shared code model | Mostly repo-local core projects | Shared `CrestApps.Core` packages plus Orchard integration modules |
| AI deployments | Embedded provider and deployment selectors | First-class typed AI deployments |
| Retrieval architecture | Mostly Azure OpenAI BYOD-centric | Provider-agnostic Data Sources, Documents, and Memory modules |
| Documentation | README-driven | Docusaurus documentation under `src\CrestApps.Docs` |

## AI deployment migration

`v2.0.0` moves AI deployments into first-class `AIDeployment` records. In the current model:

- deployments use **Purpose** to describe how the model is used
- profiles and interactions select deployments through the current deployment selectors
- tenant-wide fallbacks are managed through **Settings -> Artificial Intelligence -> Default Deployments**

After the first startup on `v2.0.0`, verify:

1. **Artificial Intelligence -> Connections** contains the expected provider connections.
2. **Artificial Intelligence -> Deployments** contains the expected deployment records and purposes.
3. **Settings -> Artificial Intelligence -> Default Deployments** points to the expected tenant defaults.
4. **Artificial Intelligence -> Profiles** still references the expected deployment names and settings.

## Breaking changes

### CancellationToken added to public async interface methods

All public async interface methods that may perform cancellable or potentially long-running work now accept an optional `CancellationToken` parameter with a default value of `default`. This is a **binary-breaking change** for implementations compiled against older versions of these interfaces. Source-level compatibility is preserved for callers because the parameter has a default value.

**What you need to do**

If you have custom implementations of any of the following interfaces, add the `CancellationToken cancellationToken = default` parameter to each overridden method:

| Interface | Methods affected |
| --- | --- |
| `IDisplayNameProvider` | `GetAsync` |
| `IUserCacheService` | `GetUserAsync`, `RemoveAsync`, `SetAsync` |
| `IAIProfileAdminMenuCacheService` | `InvalidateAsync` |
| `IFeatureSchemaProvider` | `GetFeatureIdsAsync`, `GetThemeIdsAsync` |
| `IContentSchemaProvider` | `GetPartNamesAsync`, `GetFieldTypeNamesAsync` |
| `IContentSchemaDefinition` | `GetSettingsSchemaAsync` |
| `IRecipeStep` | `GetSchemaAsync` |
| `IOmnichannelActivityStore` | `PageManualScheduledAsync`, `PageContactManualScheduledAsync`, `PageContactManualCompletedAsync`, `PageBulkManageableAsync`, `ListBulkManageableAsync`, `GetAsync` |
| `IOmnichannelActivityManager` | `PageManualScheduledAsync`, `PageContactManualScheduledAsync`, `PageContactManualCompletedAsync`, `PageBulkManageableAsync`, `ListBulkManageableAsync` |
| `IOmnichannelChannelEndpointStore` | `GetByServiceAddressAsync` |
| `IOmnichannelChannelEndpointManager` | `GetByServiceAddressAsync` |
| `IOmnichannelEventHandler` | `HandleAsync` |
| `ICampaignActionExecutor` | `ExecuteAsync` |
| `IListOmnichannelActivityFilterHandler` | `FilteringAsync` |
| `IBulkManageActivityFilterHandler` | `FilteringAsync` |

### Upgrade to .NET 10 and Orchard Core 3

`v2.0.0` requires `.NET 10` and the Orchard Core `3.0.0-preview` package line used by this repository.

**What you need to do**

1. Install the `.NET 10` SDK.
2. Move application projects to `net10.0`.
3. Update package references to the `v2` package line.
4. Build and resolve any compile-time issues in custom modules before deploying.

### Shared framework extraction to `CrestApps.Core`

Large parts of the reusable AI infrastructure are no longer implemented only inside Orchard-facing projects. The `v2` line depends on shared `CrestApps.Core` packages, while Orchard modules provide integration, UI, and runtime wiring.

**What changed**

- several Orchard-only implementation details moved into shared `CrestApps.Core.*` packages
- package references and namespace usage changed for custom code that depended on repo-local v1 internals
- samples and startup wiring follow the new package layout

**What you need to do**

1. Review custom project references.
2. Replace direct references to moved v1 implementation assemblies with the matching `CrestApps.Core` packages and namespaces.
3. Rebuild any custom modules against the new dependency graph.

### Typed AI deployments replace legacy deployment selectors

This is one of the most important breaking changes in the upgrade.

In `v1.x`, many AI settings stored provider and deployment information inline or by name. In `v2.0.0`, deployments are first-class records and related models now use typed deployment selectors such as `ChatDeploymentId` and `UtilityDeploymentId`.

**What changed**

- AI deployments are now managed as their own records
- profiles, chat interactions, and templates resolve deployments through the typed deployment model
- deployment defaults are managed centrally through site settings

**What you need to do**

1. Update recipes, appsettings, and custom code that read or write legacy deployment-name fields.
2. Review the [AI deployment migration](#ai-deployment-migration) guidance on this page.
3. After upgrade, verify that site settings contain the expected default chat deployment and other default deployment selectors.

#### Deployment connection picker and legacy alias cleanup

- the deployment editor now loads connection choices from the provider connection catalog instead of the merged `AIProviderOptions` snapshot
- provider-specific connection pickers now show the stored connections that belong to the selected provider
- deployment documents now persist the provider connection `Name` in `ConnectionName` instead of the connection record `ItemId`, and a follow-up migration normalizes older stored values
- the legacy `ConnectionNameAlias` deployment property is no longer written or used at runtime
- the deployment index backfill now also infers missing deployment types from legacy provider-connection deployment names and legacy `AIProfile` selectors such as `DeploymentId`, `ChatDeploymentId`, and `UtilityDeploymentId`, so older dictionary documents remain importable
- legacy deployment migration now merges duplicate legacy type records onto a writable deployment when possible, and creates a unique `-migrated` database record instead of mutating read-only configuration deployments when a name collision exists
- interactive legacy deployments now normalize `Chat` and `Utility` together during import, and the index-migration upgrade path now backfills default deployment site settings so upgraded tenants always receive usable chat and utility defaults

### AI Memory profile cleanup confirmation flow

The user profile AI Memory editor now keeps the **Danger zone** warning but replaces the inline clear-memory confirmation checkboxes with a standard Orchard Core admin destructive action button that triggers the confirmation dialog before deletion is processed. The action is permission-gated via an `AuthorizationHandler<PermissionRequirement>` with two permissions:

- **Clear AI memory** (`ClearAIMemory`) — allows a user to clear their own saved AI memories. The authorization handler grants this automatically when the target user matches the current user.
- **Clear AI memory for other users** (`ClearAIMemoryForOthers`) — allows administrators or authorized users to clear AI memories on behalf of other users (granted to Administrators by default).

The controller passes the target `userId` as the authorization resource, and the `AIMemoryAuthorizationHandler` determines whether the operation is for the current user (own memory) or another user, delegating to the appropriate permission check. This ensures the same authorization logic is honored system-wide.

When editing another user's profile, the messaging adapts to indicate that the action is being performed on behalf of that user.

### AI data source index backfill migration

The AI Data Sources module now adds a dedicated `AIDataSourceIndex` schema migration and backfills legacy `DictionaryDocument<AIDataSource>` records from the Orchard `Document` table into the current indexed data-source store structure, including older `CrestApps.AI` and `CrestApps.Core.AI` document type names. During import, the migration also copies a legacy `ProfileSource` value into `SourceIndexProfileName` when the newer field is missing and uses `IAIDataSourceStore.GetAllAsync()` for existing-record detection so the migration stays inside the configured store pipeline even when the map index starts empty.

The follow-up metadata migration now also infers missing source-field mappings from the selected source index profile, assigns an existing provider-matched knowledge-base index when available, and can create the default **AI Knowledge Base Warehouse** profile on demand so legacy data sources no longer fail import just because newer required index fields were absent from the original document payload.

The index-profile embedding metadata normalization step now rewrites stored embedding selectors to the deployment record `Name` instead of leaving model names such as `text-embedding-3-small` in place. This keeps knowledge-base index profiles aligned with the unique technical deployment name that runtime indexing resolves through `IAIDeploymentManager.FindByNameAsync(...)`.

The same normalization pass now reruns for tenants already migrated to the earlier version, accepts legacy Azure provider aliases such as `Azure` and `AzureOpenAIOwnData` when matching current Azure OpenAI deployments, and strips historical `Azure-` selector prefixes before rewriting the selector to the canonical deployment record name.

The follow-up rerun also covers tenants where the index-profile normalization executed before the legacy AI deployment import completed during startup. In that case, runtime indexing now repairs a unique embedding model selector such as `text-embedding-3-small` to the imported deployment record name, for example `Azure-text-embedding-3-small`, and persists the corrected selector back to the index profile.

`v2.0.0` introduces the Omnichannel platform and its related modules.

#### Subject Flows and Subject Actions

Disposition-driven behavior is no longer managed through separate Orchard Core Workflows or through the campaign editor. Instead, **Subject Flows** let you define behavior per `OmnichannelSubject` content type under **Interaction Center -> Subject Flows**.

Each subject flow owns the campaign association, interaction type, channel, and automated-channel endpoint settings for that subject. **Subject Actions** then bind dispositions to action types (**Finish**, **Try Again**, or **New Activity**) with type-specific parameters. The same disposition can trigger multiple actions. Communication preference flags (Do-Not-Call, Do-Not-SMS, Do-Not-Email, Do-Not-Chat) are available on every action type.

On the activity completion screen, selecting a disposition previews the subject actions that will execute via an AJAX call, giving users full visibility and the ability to adjust schedule dates before submitting.

**Breaking change:** The `OrchardCore.Workflows` dependency has been removed from the Omnichannel Management module. The workflow activities `CompletedActivityEvent`, `TryAgainActivityTask`, `NewActivityTask`, and `SetContactCommunicationPreferenceActivityTask` have been removed. Campaigns no longer own the disposition-driven flow configuration. Existing workflow-based or campaign-action-based setups must be migrated to subject flows and subject actions.

#### Bulk Activity Management

A new **Manage Activities** page (`Admin/omnichannel/manage-activities`) allows users with the **Manage Activities** permission to perform bulk operations on `NotStarted` manual activities. The page groups Contact and Activity filters into separate sections, supports page-size selection, shows urgency icons in activity rows, and lets managers run the six bulk actions against either the selected activities or all activities matching the current filter.

See [Omnichannel Communications](../omnichannel/index.md), [Management](../omnichannel/management.md), [Event Grid](../omnichannel/event-grid.md), and [SMS](../omnichannel/sms.md).

- missing v1 deployment fields such as `ModelName`, `ClientName`, and `Type` were normalized correctly
- default deployment site settings were backfilled
- site settings contain at least a default chat deployment when chat deployments exist

The default deployment backfill order should be verified as:

1. UI-configured default deployment
2. legacy configured default from existing settings or appsettings-backed configuration

- moved namespaces and package references
- legacy deployment-name usage
- older feature IDs
- provider-specific assumptions that no longer match the v2 module architecture

After startup on `v2.0.0`, validate:

1. AI provider connections are present.
2. AI deployments are present and usable.
3. AI profiles load correctly.
4. AI chat sessions remain available.
5. Default chat deployment and related site defaults are set correctly.
