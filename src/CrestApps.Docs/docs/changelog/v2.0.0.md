# Version 2.0.0 Release Notes

Version `2.0.0` is the major upgrade from the `v1.x` line to the current `v2` platform. It moves CrestApps.OrchardCore to Orchard Core 3 on .NET 10, expands the solution into a broader modular platform, and reshapes how AI deployments, chat experiences, retrieval, external integrations, and omnichannel communication are modeled.

This page is organized by module and feature so related changes stay together. Breaking changes, migration notes, and follow-up fixes are listed under the module or platform area they affect.

## Highlights in 2.0.0

- **Platform upgrade** to `.NET 10` and the Orchard Core `3.0.0` line used by this repository
- **Expanded AI platform** with deployments, provider connections, orchestrators, tools, AI Documents, AI Memory, MCP, A2A, Claude, and Copilot integrations
- **Broader Orchard module surface** including Content Access Control, Content Transfer, phone-number support, DNC Registry, Omnichannel Management, Recipes, SignalR, Roles, and Time Zones
- **Provider-agnostic retrieval architecture** built around AI Data Sources, AI Documents, and AI Memory instead of provider-bound BYOD flows
- **Canonical Docusaurus documentation** under `src\CrestApps.Docs`

## Recommended migration path

Use this order when moving a tenant or solution from `v1.x` to `v2.0.0`:

1. Back up every tenant database and export any important recipes or deployment plans.
2. Install the `.NET 10` SDK and move application projects to `net10.0`.
3. Update package feeds and package references to the `v2.0.0` line.
4. Review custom code for moved namespaces, shared `CrestApps.Core` packages, and updated async interface signatures.
5. Start the application once and allow the data migrations to complete.
6. Review AI provider connections, AI deployments, default deployment settings, AI profiles, and chat data in the admin UI.
7. Rebuild and validate tenant features such as AI chat, data sources, documents, MCP, Omnichannel imports, and custom modules.

## Before you upgrade

Before upgrading from `v1.x` to `v2.0.0`, back up every tenant database and review your custom code, package references, enabled features, and configuration.

At a high level, the platform changes are:

| Area | v1.x | v2.0.0 |
| --- | --- | --- |
| .NET SDK | `8.0.100` | `10.0.100` |
| Target frameworks | `net8.0;net9.0` | `net10.0` |
| Orchard Core | `2.1.x` line | `3.0.0-preview` line |
| NuGet feeds | NuGet only | NuGet + Orchard Core preview + CrestApps Core preview |
| Shared code model | Mostly repo-local core projects | Shared `CrestApps.Core` packages plus Orchard integration modules |
| AI deployments | Embedded provider and deployment selectors | First-class typed deployment records |
| Retrieval architecture | Mostly Azure OpenAI BYOD-centric | Provider-agnostic Data Sources, Documents, and Memory modules |
| Documentation | README-driven | Docusaurus documentation under `src\CrestApps.Docs` |

## Module and feature changes

### Artificial Intelligence

#### Core platform

- The AI platform now centers around first-class deployment records, provider connections, centralized defaults, and more source-agnostic editors and settings.
- `v2.0.0` replaces older inline or name-based deployment selectors with typed deployment records. Profiles, chat interactions, templates, and related settings now resolve deployments through the current deployment model.
- Deployment migration now imports legacy `DictionaryDocument<AIDeployment>` records, infers missing deployment types from legacy selectors, backfills default deployment site settings, merges duplicate legacy records when possible, and creates unique migrated records instead of mutating read-only configuration deployments.
- Deployment editors now load connection choices from the provider-connection catalog, provider-specific pickers show only the connections that belong to the selected provider, and deployment documents now persist the provider connection `Name` in `ConnectionName`.
- The deployment model now uses **Purpose** instead of **Type**. The admin UI reflects that rename, the migration persists converted values through `UpdateFrom6`, and a new `Vision` purpose is available in the default deployment settings and chat surfaces.
- The shared Orchard-facing AI code now uses `IClock` consistently for timestamps, and the Orchard integration now defers to the shared `CrestApps.Core.AI.Handlers.AIProfileHandler` instead of keeping duplicate local profile validation handlers.
- AI administration is reorganized in the v2 line. Settings, profile flows, and deployment defaults are grouped differently than in `v1.x`, and operator guidance should be updated accordingly.
- The AI configuration section now uses the nested `OrchardCore:CrestApps:AI` path. The old `CrestApps_AI` paths still work, but they log deprecation warnings and should be treated as transitional.

#### A2A

- `v2.0.0` adds Agent-to-Agent support so Orchard-hosted AI solutions can participate in broader multi-agent workflows.
- A2A host authentication now defaults to `OpenId` when no authentication type is configured, so anonymous access must be chosen explicitly.
- Shared OAuth 2.0 editors in the A2A SSE configuration paths now render `Token Endpoint`, `Client ID`, and `Scopes` only once, which preserves posted values and avoids validation failures during save.
- See [A2A](../ai/a2a/index.md).

#### Agent

- The AI platform now supports orchestrator-based execution, allowing tools, retrieval, templates, agents, and external integrations to participate in coordinated flows instead of relying on a simpler prompt-routing model.
- The `createOrUpdateContentItem` tool now resolves corrective payload guidance and payload validation through a swappable service, keeps the dropped-value detection after `ContentItem` mapping, and can use recipe-backed content-type schemas when the Recipes feature is enabled.
- AI content-item tools now instruct models to create the top-level content item once and embed nested `BagPart`, `FlowPart`, and widget-style child items in the same payload instead of issuing separate tool calls.
- Recipe-backed AI content-item validation now evaluates the authored schema as-is, helper-tool dependencies can be declared during tool registration, and the AI tooling automatically includes `getContentItemSchema` and `getOrchardCoreRecipeJsonSchema` where appropriate.
- The `getContentItemSchema` and `getOrchardCoreRecipeJsonSchema` flows now expose richer schema discovery, preserve the root `steps` array behavior, keep the full step-name enum available, and report valid step names when an unknown step is requested.
- See [Artificial Intelligence Suite](../ai/index.md), [AI Agent](../ai/agent.md), and [AI Tool Management](../ai/tools.md).

#### Chat

- The base AI chat experience is broader in `v2.0.0`, with richer session analytics, improved admin and frontend widget scenarios, more deployment-aware and profile-aware flows, and stronger post-session processing support.
- The AI chat admin UI, admin widget, frontend widget, and chat interaction surfaces now render `[doc:N]` markers as superscript citations with linked reference lists when the underlying references can be resolved.
- Orchard-hosted chat surfaces now use the rebuilt bundled scripts and styles from the repository instead of older pinned CDN artifacts.
- Playback and conversation controls only appear when the required speech deployments are actually available for the tenant.
- The **Prompt subject** editor only appears for **Template generated prompt** profiles and profile-source templates.
- The frontend widget now removes theme-added paragraph bottom spacing inside rendered messages so wrapped `<p>` blocks do not create extra gaps.
- The shared `list-management-ui` script now registers `window.listManagementUI` explicitly so admin catalog pages such as **Chat Interactions** initialize reliably.
- The admin AI chat session page now keeps the placeholder element rendered even when a historical session already has messages, which prevents the shared chat UI runtime from aborting initialization when users reopen an existing session from history.
- See [AI Chat](../ai/chat.md) and [AI Chat Analytics](../ai/chat-analytics.md).

#### Chat.Claude

- `v2.0.0` adds Claude support as a first-class Orchard integration with provider-specific settings and Claude-oriented deployment and model selection support.
- See [Claude](../ai/claude.md).

#### Copilot Integration

- The new Copilot integration brings a GitHub Copilot-style chat experience into Orchard Core while still allowing Orchard-specific tools, content access, and orchestration behavior.
- Copilot site settings now include a **Not configured** authentication mode so administrators can disable the orchestrator without deleting previously stored credentials.
- GitHub OAuth client ID and client secret validation only run when that authentication mode is selected, and an existing stored secret remains valid until it is explicitly replaced.
- The **Settings -> Artificial Intelligence** page now keeps the memory guidance in the matching **Memory** section context, and the AI settings cards render in a stable order so **Admin Widget**, **Copilot**, and **Claude** appear together predictably.
- See [AI Chat Copilot Integration](../ai/copilot.md).

#### Chat.Interactions

- `v2.0.0` adds ad-hoc chat interactions that do not require a predefined AI profile for every scenario.
- These interactions support more dynamic admin and frontend AI experiences plus richer file, image, and interaction-driven workflows.
- Chat interaction surfaces now respect the current document-upload and image-upload settings so each UI exposes only the extensions allowed for the tenant and profile.
- See [AI Chat Interactions](../ai/chat-interactions.md).

#### Data Sources

- AI Data Sources are now a first-class, provider-agnostic capability instead of an Azure OpenAI BYOD-only story.
- The Orchard integration now bridges `IDocumentIndexHandler` source-index notifications into an Orchard-managed `IAIDataSourceIndexingQueue`, batching request-scoped work and dispatching it after the request so source-index changes refresh mapped knowledge-base indexes automatically.
- Orchard-specific queued sync work now overrides `IAIDataSourceIndexingService` correctly before background work is dispatched, which keeps re-syncs on the Orchard adapter path.
- Synchronizing a source index profile now queues a full re-sync for every mapped AI data source, and the knowledge-base profile is resolved consistently by profile `Name` during that work.
- The AI Data Sources module now adds a dedicated `AIDataSourceIndex` migration that backfills legacy `DictionaryDocument<AIDataSource>` records, supports older `CrestApps.AI` and `CrestApps.Core.AI` type names, repairs missing `ProfileSource` values, infers source-field mappings, assigns or creates knowledge-base profiles as needed, and normalizes embedding selectors to the canonical deployment record names.
- The normalization pass now also accepts legacy Azure provider aliases, strips old `Azure-` prefixes where needed, and repairs tenants where deployment import completed after earlier index-profile normalization.
- The Data Sources UI now groups **Source Index** and **Knowledge Base Index** choices under localized provider names, filters the **Source Index** selector to exclude AI-managed index types, and fixes name filtering so display-text searches behave correctly.
- See [AI Data Sources](../ai/data-sources/index.md).

#### AI Documents

- `v2.0.0` adds a dedicated AI document pipeline for upload, parsing, chunking, indexing, and retrieval.
- The document story now includes PDF and OpenXML support plus dedicated storage and indexing integrations.
- Legacy AI documents stored under earlier `CrestApps.OrchardCore.AI.*` or `CrestApps.AI.*` document type names are rewritten to the current shared model names so older uploaded profile and session documents remain readable and removable.
- AI Documents settings now expose the default **Document retrieval mode**, and the document-related profile, template, and interaction editors use localized retrieval-mode choices.
- Managed mapping normalization now keeps built-in Azure AI Search document index fields such as `Content` and `Embedding` from being duplicated repeatedly on later edits.
- See [AI Documents](../ai/documents/index.md), [PDF support](../ai/documents/pdf.md), [OpenXML support](../ai/documents/openxml.md), and [Azure Blob Storage](../ai/documents/azure-blob-storage.md).

#### MCP

- The platform now supports both exposing Orchard Core as an MCP server and consuming external MCP servers as a client.
- `v2.0.0` also expands the MCP surface with FTP and SFTP resource modules.
- MCP API key authentication now uses constant-time secret comparison.
- MCP host authentication now defaults to `OpenId` when no authentication type is configured, so anonymous access must be chosen explicitly.
- Shared OAuth 2.0 editors in MCP SSE configuration paths now render `Token Endpoint`, `Client ID`, and `Scopes` only once, which preserves posted values and avoids validation failures during save.
- See [Model Context Protocol](../ai/mcp/index.md), [MCP Server](../ai/mcp/server.md), [MCP Client Integration](../ai/mcp/client.md), [FTP MCP resources](../ai/mcp/ftp.md), and [SFTP MCP resources](../ai/mcp/sftp.md).

#### Memory

- `v2.0.0` adds persistent AI memory support so assistant behavior can retain context across sessions and use provider-backed retrieval or indexing strategies.
- The user-profile memory editor now uses Orchard Core's standard destructive confirmation flow when clearing saved memory and relies on permission-aware authorization for self-service and delegated clearing.
- Managed mapping normalization now keeps built-in Azure AI Search AI Memory fields from being duplicated on later edits.
- The AI Memory index-profile editor now treats the embedding deployment as a creation-time choice only, showing it as read-only for existing profiles because migrations normalize older data ahead of time.
- See [AI Memory](../ai/memory.md).

#### Prompting

- Prompt templates and profile templates are now first-class reusable building blocks in the AI platform.
- This makes tenant-level AI experiences more reusable and more consistent across chat, interactions, and automation scenarios.

#### Response handlers

- Chat handling is now extensible through pluggable response handlers.
- This enables AI-driven responses, deferred external processing, live-agent handoff, and hybrid AI and human support scenarios.
- See [Response Handlers](../ai/response-handlers.md) and [Chat UI Notifications](../ai/chat-notifications.md).

### Content Access Control

#### Role-based content restrictions

- `v2.0.0` includes the `CrestApps.OrchardCore.ContentAccessControl` feature as part of the broader module surface.
- The feature lets administrators restrict access to content items by attaching `RolePickerPart` to a content type and requiring one or more roles for access.
- Access control is enabled through `RolePickerPartContentAccessControlSettings`, including the `Restrict content?` setting that turns on enforcement for the attached content type.
- The release also adds recipe-schema coverage for `RolePickerPartContentAccessControlSettings`, which keeps content-definition and recipe-based provisioning aligned with the feature.
- Content Access Control admin editors now use the current Orchard admin field layout so labels, wrappers, and validation stay consistent with the rest of the v2 admin experience.
- See [Content Access Control](../modules/content-access-control.md).

### Content Fields

#### Phone Numbers

- `v2.0.0` adds `CrestApps.OrchardCore.ContentFields` with a `PhoneField` that stores phone numbers in E.164 format together with the ISO country code and national number.
- The editor uses `intl-tel-input` for country-aware entry and validates values through `IPhoneNumberService`.
- Omnichannel Management now migrates `PhoneNumberInfoPart.Number` from a `TextField` to a `PhoneField`.

### Content Transfer

#### Core transfer

- `v2.0.0` adds `CrestApps.OrchardCore.ContentTransfer`, `CrestApps.OrchardCore.ContentTransfer.Core`, and `CrestApps.OrchardCore.ContentTransfer.Abstractions`.
- CSV support stays in the base Content Transfer feature, while optional Excel workbook support moves into `CrestApps.OrchardCore.ContentTransfer.OpenXml`.
- Content-type settings now default **Allow Bulk Import** and **Allow Bulk Export** to enabled so types participate by default and can explicitly opt out.
- Import and export content-type selectors are now sorted alphabetically to make the bulk-transfer UI easier to scan.
- Imports now commit inline status changes before deferred background jobs run and use **Pending**, **Paused**, and **Deleting** states in the admin list.
- The admin actions now use **Pause import** and **Resume import** instead of the earlier cancel or process wording, and older canceled rows migrate to the paused state.
- Bulk imports now include **Publish imported content**, letting administrators choose between saving a latest draft and publishing immediately after create or update.
- Updates targeting an existing `ContentItemId` now apply imported values onto a new latest version and then save or publish that version according to the selected option.
- `ContentItemVersionId` is now treated as an export-only value for versionable content types and is ignored during import.
- The export pipeline now welds lazily created parts onto the parent content item during column discovery, import, and export so `.xlsx` exports no longer fail with a JSON node-cycle exception when a configured part is not yet materialized.
- The module now includes unit coverage for imported content-transfer services and Excel validation behavior, plus updated module documentation.

#### Omnichannel contact import and export integration

- Omnichannel contact imports and exports now map the first managed contact-method entries to workbook `Email`, `Cell Phone`, and `Phone` columns.
- The pipeline now reads and writes `DoNotCall`, `DoNotCallUtc`, `DoNotSms`, `DoNotSmsUtc`, `DoNotEmail`, `DoNotEmailUtc`, `DoNotChat`, and `DoNotChatUtc` on `OmnichannelContactPart`.
- The import metadata now advertises `true` and `false` for the `DoNot*` columns.
- Duplicate-phone filtering is enabled by default, skipped duplicate rows are added to the error export with a reason, and duplicate detection checks both the current import batch and existing Orchard contact records.
- Imports fall back to older stored phone values when upgraded tenants have not yet backfilled the newer normalized-phone indexes.
- When a row includes an existing `ContentItemId`, matching phone numbers on that same content item are treated as updates instead of duplicate conflicts.
- Omnichannel contact imports now require a single default country for non-E.164 phone numbers, normalize contact methods before duplicate and DNC checks, preserve existing E.164 values, and replace managed `Email`, `Cell`, and `Home` contact-method entries during updates so re-imports do not keep appending stale values.
- Omnichannel exports now include `TimeZoneId`, and imports can infer missing time-zone values from imported phone numbers through `IPhoneNumberService`.

#### Core platform

- `v2.0.0` introduces the Omnichannel platform together with management, Event Grid, and SMS integrations.
- `OmnichannelContactPart` now supports **Require time zone** and opt-in rendering of the Do not SMS, Do not chat, and Do not email preference fields, with **Require time zone** and **Use Do not call** enabled by default.

- Omnichannel Management upgrade migrations now rerun both the default-contact-index repair and the legacy `PhoneNumberInfoPart.Number.Text` backfill for upgraded tenants, execute the phone rewrite through a direct deferred migration task, and reparse older North American lead numbers into `PhoneField` storage with a migration-only fallback.

#### Event Grid

- Automated subject flows now select a chat AI profile instead of configuring provider, connection, deployment, prompt, and tuning settings inline.

- Omnichannel dispositions now use unique immutable names after creation, the delete action is wired correctly, activity batches no longer ask for a campaign, and user pickers now use the named user-search route instead of hard-coded admin API paths.
- Editing a completed omnichannel activity now suppresses workflow preview UI and does not imply that disposition changes will create new follow-up work.
- A new **Manage Activities** page (`Admin/omnichannel/manage-activities`) supports page-size selection, separate Contact and Activity filter groups, urgency icons, and six bulk actions against selected or filtered manual activities.
- Admin editors in Omnichannel Management now use the current `ocat-*` admin layout classes instead of older generic Bootstrap field markup.

#### SMS

- `v2.0.0` adds SMS channel support as part of the Omnichannel platform.
- The Twilio webhook now reuses the parsed form payload instead of reading the request body twice.

### Phone Number Verifications

#### Provider-agnostic verification framework

- `v2.0.0` introduces the **Phone Number Verifications** module (`CrestApps.OrchardCore.PhoneNumbers.Verifications`), a provider-agnostic framework for verifying contact phone numbers. The core feature is enabled on demand and does not depend on any external verification provider.
- The core feature now declares a dependency on `OrchardCore.Contents`, since it defines the `PhoneNumberVerificationPart` content part and a migration that requires `IContentDefinitionManager`. Enabling the feature on a tenant without content management previously failed to activate the shell.
- Shared phone-number services, constants, and verification index types now live in `CrestApps.OrchardCore.PhoneNumbers.Core`, so the Phone Numbers and Phone Number Verifications modules use the same core infrastructure.
- Verification providers are pluggable: they implement `IPhoneNumberVerificationProvider` from `CrestApps.OrchardCore.PhoneNumbers.Abstractions`, are registered against a unique provider key through `AddPhoneNumberVerificationProvider`, and configure localized display names and descriptions through the registration action. Providers are discovered dynamically so adding a provider never requires changes to the core feature.
- Provider HTTP payload parsing now uses shared snake-case JSON serializer options instead of per-property DTO attributes, keeping provider response models simpler and easier to maintain.
- The provider selection setting lists every enabled provider, and the framework resolves providers by key through a lean `IPhoneNumberVerificationManager` without a provider ever selecting itself.
- Providers map their native responses into a shared `PhoneNumberVerificationResult` model that includes validity, reachability, line type, country, location, carrier, time zone, line status, risk details, the raw provider response, and an extensible metadata bag.

#### Contact storage, indexing, and reporting

- Verification data is stored directly on content items through the new `PhoneNumberVerificationPart`, which retains the submitted and normalized phone number, full normalized provider response, last verification, verifying user id, attempt count, and next due date. User identifiers are stored instead of usernames because usernames can change over time.
- Omnichannel contacts now trigger phone-number verification automatically after create or update when their preferred phone number is new or changed. The handler uses deferred work after the content item is saved, and stores the number as `Unverified` if no provider is currently available.
- A `PhoneNumberVerificationPartIndex` SQL index exposes the commonly queried fields for reporting, dashboard widgets, revalidation jobs, and administrative searches, while provider-specific metadata stays in the stored JSON payload.
- A report dashboard under **Reports** -> **Phone Number Verifications** surfaces operational metrics such as verified, unverified, invalid, mobile, landline, and VoIP counts, numbers pending verification or requiring revalidation, the verification success rate, failures, and provider usage. Access to the dashboard is controlled by the dedicated `RunPhoneNumberVerificationsReport` permission.

#### Revalidation and cost optimization

- A daily background task revalidates content items that already carry a stored phone number and whose verification has expired, processing work in resilient batches with distributed locking so it scales and tolerates provider failures.
- Verification data on `PhoneNumberVerificationPart` acts as the authoritative cache: numbers are only (re)verified when expired or explicitly requested, which minimizes calls to paid provider APIs.

#### AbstractAPI provider

- `v2.0.0` ships the **AbstractAPI Phone Number Verification** feature (`CrestApps.OrchardCore.PhoneNumbers.Verifications.AbstractApi`), which verifies numbers through the AbstractAPI Phone Validation service and maps the response into the common result model.
- AbstractAPI settings appear on a dedicated tab within the Phone Number Verifications settings page, following the Orchard Core SMS provider settings pattern, and the API key and other secrets are stored as protected values.

#### Veriphone provider

- `v2.0.0` ships the **Veriphone Phone Number Verification** feature (`CrestApps.OrchardCore.PhoneNumbers.Verifications.Veriphone`), which verifies numbers through the Veriphone `/v2/verify` endpoint and maps the response into the common result model.
- Veriphone settings appear on a dedicated tab within the Phone Number Verifications settings page. The API key is stored as a protected value and sent to Veriphone with the recommended `Authorization: Bearer` header.

#### Twilio provider

- `v2.0.0` ships the **Twilio Phone Number Verification** feature (`CrestApps.OrchardCore.PhoneNumbers.Verifications.Twilio`), which verifies numbers through the Twilio Lookup v2 API and maps the response into the common result model.
- Twilio settings appear on a dedicated tab within the Phone Number Verifications settings page. The provider supports Twilio's API key SID/API key secret authentication and Account SID/Auth Token authentication, with secrets stored as protected values.
- The Twilio provider supports optional Lookup data packages through the `Fields` setting so administrators can request line type intelligence, line status, SMS pumping risk, and other paid add-ons when needed.
- Fixed the Twilio settings tab failing to render (a truncated response and `NS_ERROR_NET_PARTIAL_TRANSFER` in the browser): the localized endpoint hint containing a literal `{PhoneNumber}` token is now escaped so it is no longer treated as a composite format placeholder.

### Platform and tooling

#### Build, packaging, and shared runtime

- `v2.0.0` requires `.NET 10` and the Orchard Core `3.0.0-preview` package line used by this repository.
- Large parts of the reusable runtime moved out of Orchard-only projects and into shared `CrestApps.Core.*` packages. Custom modules that depended on older repo-local implementation assemblies should be rebuilt against the new package layout.
- Public async interfaces that can perform cancellable or longer-running work now accept an optional `CancellationToken cancellationToken = default` parameter. Custom implementations must update their signatures to match.
- The checked-in `NuGet.config` now restores Orchard Core packages from the Orchard Core preview feed and `CrestApps.Core*` packages from the CrestApps Core preview feed.
- Package tags are now computed centrally during packing so NuGet metadata is more consistent across the solution.

#### Frontend and workspace maintenance

- The root npm workspace now pins patched transitive versions of `lodash`, `yaml`, and `brace-expansion` through `overrides`, which removes the shared asset-build audit warnings.
- The repository `gulpfile.js` now waits for CSS and JavaScript output streams to finish before reporting task completion, which keeps `npm run rebuild` stable on current Node.js and Gulp versions.

### Recipes

#### Core recipe and schema platform

- `v2.0.0` expands the recipe surface, the surrounding samples, and the schema-export tooling.
- The recipe `content` step now enumerates `ContentType` values from tenant definitions, contributes content-type-specific part and field hints, exposes typed built-in field payloads, and keeps common metadata properties available for every item.
- Recipe content schemas now model built-in part payloads and additional Orchard settings more completely, including common body parts, title and autoroute payloads, scheduling settings, list settings, layout settings, HTML settings, and nullable built-in field payload values for standard Orchard field types.
- `ContentDefinition` and `ReplaceContentDefinition` schemas now register built-in Orchard Core field schema contributors such as `TextFieldSettings`, `MediaFieldSettings`, `TaxonomyFieldSettings`, `GeoPointFieldSettings`, and `MarkdownFieldSettings`.
- The Recipes module now also exposes the Orchard Core `move-attached-media-fields` step schema, validates `Roles.Permissions[]` entries against the permission enum correctly, omits placeholder permission values from generated role schemas, and treats `ContentDefinition` as valid when either `ContentTypes` or `ContentParts` is supplied.
- Recipe content-item schema composition now flows through `IContentItemSchemaService`, replacing the older `ContentCommonSchemas` helper.
- Recipe content schemas now understand `OmnichannelContactPart.TimeZoneId`, the E.164 `PhoneNumberInfoPart.Number.PhoneNumber` payload, `PhoneField` settings and payloads, and CrestApps schema contributors for parts such as `AIProfilePart`, `RolePickerPart`, `RolePickerPartContentAccessControlSettings`, and `UserFullNamePart`.
- The generic `Settings` schema now includes `LoginSettings.AllowRememberMe`, DNC registry site settings, and feature-gated contributed settings fragments for base settings such as `HomeRoute` as well as AI and user-profile settings.
- Deployment-plan steps now use localized admin categories, grouping AI, AI Data Sources, and MCP export steps under **Artificial Intelligence** and `TimeZoneMaps` under **Infrastructure**.
- The recipe schema exporter now instantiates service-backed schema steps such as `AdminMenu` instead of requiring parameterless constructors, and the `CrestApps.OrchardCore.RecipeSchemaExporter` utility now registers default `IOptions<AIOptions>` so AI-backed steps can be materialized successfully.

#### AI and contributor guidance

- AI recipe schemas now enumerate known nested metadata objects for `AIProfileTemplate`, `AIDataSource`, `McpConnection`, `McpPrompt`, `McpResource`, and `A2AConnection` while still allowing extension keys.
- The `AIProfile` recipe step now matches the current import and export contract, including the `Agent` profile type, prompt subject, orchestrator name, deployment selectors, and metadata fields while excluding obsolete selectors such as `ChatDeploymentId` and `UtilityDeploymentId`.
- The `AIProfile` and `CreateAIProfileFromTemplate` schemas now describe the known objects available under `Properties` and `Settings`, and `AIProfileTemplate` reuses that same known object catalog.
- Built-in CrestApps recipe-step schemas now provide richer per-property descriptions across steps such as Roles, Workflows, Media, Layers, Placements, Templates, Features, and related admin and import steps.
- The Recipes documentation now explains when to implement `PartSchemaDefinitionBase`, `FieldSchemaDefinitionBase`, and `IContentSchemaDefinition`, shows both part and field examples, and clarifies how multiple schema contributors for the same part or field name are merged.
- The Recipe Schema Exporter utility now lives under `utilities\` instead of `tests\`, and the documented build and run commands were updated accordingly.

### Time Zones

#### Time zone maps

- `v2.0.0` adds the `CrestApps.OrchardCore.TimeZones` module, which manages friendly named time-zone maps through admin UI, recipes, deployment plans, and a seed migration recipe.
- Omnichannel time-zone selectors now read from `ITimeZoneSelectListProvider` instead of Orchard Core's raw IANA list.
- The Time Zones admin list now shows the mapped `TimeZoneId` as a badge, renders the author through the shared `user-display-name` summary badge, and preserves `Author`, `OwnerId`, `CreatedUtc`, and `ModifiedUtc` metadata through recipe and deployment import and export.
- The Time Zones docs now clarify that the module overrides Orchard Core's `ITimeZoneSelectListProvider`, and the seed recipe now stamps admin-owned audit metadata through shared recipe variables.

## Migration from v1.x

### Step 1: Back up tenant data

Back up every tenant database before first startup on `v2.0.0`.

This is especially important for tenants that already contain:

- AI provider connections
- AI deployments
- AI profiles
- AI chat sessions

### Step 2: Update runtime, packages, and feeds

1. Install `.NET 10`.
2. Update to the `v2.0.0` package line.
3. Move projects to `net10.0`.
4. Adopt the v2 `NuGet.config` and preview package feeds.

### Step 3: Re-map enabled features

Review the v1 modules and provider-specific features you enabled and map them to the current v2 module set.

### Step 4: Migrate AI deployments and defaults

The v2 upgrade path must preserve legacy AI connections, deployments, profiles, and chat-session data from real `v1.x` tenant databases.

During deployment migration, verify:

- legacy `DictionaryDocument<AIDeployment>` records were imported
- missing v1 deployment fields such as `ModelName`, `ClientName`, and `Type` were normalized correctly
- default deployment site settings were backfilled
- site settings contain at least a default chat deployment when chat deployments exist

The default deployment backfill order should be:

1. UI-configured default deployment
2. legacy configured default from existing settings or appsettings-backed configuration
3. the first available deployment of that type

### Step 5: Review custom code

Review custom modules for:

- moved namespaces and package references
- legacy deployment-name usage
- older feature IDs
- provider-specific assumptions that no longer match the v2 module architecture
- interface implementations that now need `CancellationToken` parameters

### Step 6: Validate the upgraded tenant

After startup on `v2.0.0`, validate:

1. AI provider connections are present.
2. AI deployments are present and usable.
3. AI profiles load correctly.
4. AI chat sessions remain available.
5. Default chat deployment and related site defaults are set correctly.
6. Retrieval, document, MCP, Content Transfer, DNC, Omnichannel, Recipes, and Time Zone features still map to the intended modules and behaviors.
